## 0. อ่านก่อน — เอกสารนี้ให้ review อะไร
แพรขอให้ผมยืนยันว่า **"อ่านข้อมูลมาถูก"** ก่อนสร้างจริง — 3 จุดที่ต้องเช็ก (มีตารางในข้อ 7):

2. **Par Stock** — เข้าใจว่าเป็น "ยอดขั้นต่ำที่ต้อง reserve ต่อสาขา เติมสัปดาห์ละ 2 ครั้ง (พุธ+เสาร์)" ถูกไหม
3. **ฟอร์มกรอกสต็อกสิ้นวัน** — mock screen สำหรับพนักงานกรอกผ่าน iPad/มือถือ ตรงที่ต้องการไหม

## 1. Problem Statement
ตอนนี้ร้านจัดการสต็อก 2 สาขา (SND, NVP) บน Excel + สคริปต์ Python ที่ต้องรันเอง — พนักงานกรอกยาก, มี manual step (copy→paste values), path สคริปต์ผูกกับ sandbox รันบนเครื่องจริงไม่ได้, และไฟล์ xlsx พังง่ายบน Excel Mac. ผลคือ **ข้อมูลสต็อกไม่ real-time, เสี่ยงกรอกผิด, และเจ้าของต้องลงมือเองทุกวัน** ยิ่งขยายสาขายิ่งไม่ scale.

## 2. Goals (วัดผลได้)
1. พนักงานกรอกสต็อกสิ้นวันจบใน **≤ 5 นาที/สาขา** ผ่าน iPad/มือถือ (จากเดิมต้องเปิด Excel + รอรันสคริปต์)
2. ตัด manual step ทั้งหมด — **0 ครั้ง** ที่ต้อง copy/paste หรือรันสคริปต์ด้วยมือ
3. ยกมา (carry-forward) + คำนวณคงเหลือ + Variance **อัตโนมัติ 100%** ไม่ต้องพิมพ์ซ้ำ
4. รอบเติมของ (พุธ/เสาร์) เห็น **"ต้องเติมกี่ชิ้นให้ถึง Par"** ต่อสาขาทันที ลดของขาด/ของเกิน
5. เจ้าของเห็นสต็อก + ยอดขายทุกสาขาแบบ real-time โดยไม่ต้องเปิดไฟล์

## 3. Non-Goals (นอกขอบ v1 — กัน scope creep)
1. **ต้นทุน-กำไรต่อเมนู / ตัดสต็อกอัตโนมัติจากยอดขายผ่าน BOM** — v1 เก็บ BOM ไว้ review เฉย ๆ ยังไม่ auto-deduct (ซับซ้อน แยกเป็น Phase 2)
2. **ระบบสั่งซื้อ/PO ส่งซัพพลายเออร์** — v1 แค่บอก "ต้องเติมเท่าไร" ไม่ยิงออเดอร์จริง
3. **ตารางพนักงาน / เงินเดือน** — คนละโมดูล
4. **POS / ตัดเงินหน้าร้าน** — v1 บันทึกยอดขายรวมรายวันแบบ manual เหมือนเดิม ไม่เชื่อม POS
5. **แอปมือถือ native (App Store/Play)** — v1 เป็นเว็บ responsive เปิดผ่านเบราว์เซอร์ก่อน

## 4. Target Users
- **พนักงานหน้าร้าน (คนกรอก)** — กรอกสต็อกสิ้นวัน + ยอดขาย ผ่าน iPad/มือถือ ต้องเร็ว กดง่าย ไม่งง
- **แพร (เจ้าของ)** — ดูภาพรวมสต็อก/ยอดขายทุกสาขา, เช็กของใกล้หมด, วางแผนเติมของ
- **ผู้ช่วยหลังบ้าน/AI Analyst** — วิเคราะห์ trend, forecast การสั่งของ (ต่อยอดจาก YC Stock Analyzer)

## 5. User Stories
**พนักงานหน้าร้าน**
- ในฐานะพนักงาน ฉันอยากเลือกสาขา+วันที่ แล้วกรอก "ขายไป/ใช้ไป" กับ "คงเหลือ" ของแต่ละรายการบนมือถือ เพื่อปิดสต็อกสิ้นวันได้เร็ว
- ในฐานะพนักงาน ฉันอยากให้ช่อง "ยกมา" ขึ้นให้อัตโนมัติจากคงเหลือเมื่อวาน เพื่อไม่ต้องเปิดไฟล์เก่าหา
- ในฐานะพนักงาน ฉันอยากเห็นทันทีถ้ากรอกแล้วยอดไม่ตรง (Variance แดง) เพื่อแก้ก่อนส่ง
- ในฐานะพนักงาน ฉันอยากกรอกเป็นหมวด ๆ (Yogurt, Toppings, ถ้วย…) เพื่อไม่สับสนกับ 114 รายการรวด

**แพร (เจ้าของ)**
- ในฐานะเจ้าของ ฉันอยากเห็นรายการที่คงเหลือ < Par ของแต่ละสาขา เพื่อรู้ว่ารอบพุธ/เสาร์ต้องเติมอะไรบ้าง
- ในฐานะเจ้าของ ฉันอยากเห็น "ต้องเติมกี่ชิ้น" (Par − คงเหลือ) ต่อสาขา เพื่อเตรียมของได้พอดี
- ในฐานะเจ้าของ ฉันอยากดูยอดขายรายวันแยกช่องทาง (เงินสด/QR/EDC/Grab/Lineman) ต่อสาขา
- ในฐานะเจ้าของ ฉันอยากเทียบคงเหลือวันนี้กับเมื่อวาน เพื่อจับความผิดปกติ

## 7. สรุปโครงสร้างข้อมูล — ให้แพร REVIEW ✅

### 7.1 Data Model (entity หลัก)
```
Branch (สาขา)            : SND, NVP  (เพิ่มได้)
Item (รายการสต็อก)       : id, ชื่อ, หมวด, หน่วยบรรจุ, is_special, is_cup, cup_size,
                           has_remainder (แกะ), grams_per_uom (จำนวน/แพค),
                           remainder_group (กลุ่มเศษรวม), sort   (112 รายการ · config แก้ได้หน้า Settings)
ParLevel (ขั้นต่ำ/สาขา)   : item × branch → จำนวนขั้นต่ำ (null = "-" ไม่ stock)
StockDaily (สต็อกสิ้นวัน) : วันที่ × สาขา × item → ยกมา, รับเข้า, ขาย/ใช้, ส่งคืน/เสีย,
                           คงเหลือ(แพ็ค+เศษ), หมายเหตุ, variance   (unique วันที่|สาขา|item)
SalesDaily (ยอดขาย/วัน)   : วันที่ × สาขา → เงินสด, QR, EDC, Grab, Lineman, รวม
CupReconcile (ถ้วยเสิร์ฟ)  : วันที่ × สาขา → ต่อขนาด(P/S/Bowl/14oz): ขายจริง (กรอก);
                           ตั้งต้น/รับเข้า/คงเหลือ ดึงจาก StockDaily ของถ้วย (แพ็ค×50 + เศษ)
Menu + BOM (สูตรเมนู)     : menu → [ingredient, qty, unit]  (v1: reference เท่านั้น)
```

### 7.2 ✅ REVIEW #1 — สูตรเมนู (BOM) ผมถอดมาถูกไหม
BOM มี **7 หมวด** โครงเดียวกัน: *เมนู → รายการวัตถุดิบ + จำนวน + หน่วย* — ตัวอย่างหมวดละ 1 เมนู (ตารางเต็มทุกเมนูดูใน [`review.html`](review.html)):

**หมวด YOGURT CREATIONS** (โบว์ล ฐาน Greek Yogurt 150g)
| เมนู | วัตถุดิบ | จำนวน | หน่วย |
|---|---|---|---|
| Crunch Berry Bliss | Greek Yogurt | 150 | g |
| | Strawberry | 20 | g |
| | Blueberry | 20 | g |
| | Granola | 20 | g |
| | Honey | 15 | g |

### 7.3 ✅ REVIEW #2 — Par Stock (ยอด reserve + รอบเติม พุธ/เสาร์)
**เข้าใจตามที่แพรบอก:** Par = **ยอดขั้นต่ำที่ต้องมี (reserve) ต่อสาขา** · เติมของ **สัปดาห์ละ 2 ครั้ง (ทุกวันพุธ + เสาร์)** · แต่ละรอบต้องเติมให้สต็อกกลับมา **≥ Par**

**ตรรกะที่ระบบจะใช้:**
```
ต้องเติม (รอบพุธ/เสาร์) = MAX(Par(สาขา) − คงเหลือปัจจุบัน, 0)
```

- รายการอื่น (~103) = เข้าได้ **ทั้งพุธ + เสาร์** ตามปกติ
- ผลกับระบบ: หน้า "ต้องเติม" ต้อง **กรองตามสาขา × วัน** — เช่น NVP รอบ **เสาร์** จะไม่โชว์ 7 รายการนี้ (เพราะมันเข้าวันพุธ), SND รอบ **พุธ** ก็ไม่โชว์เช่นกัน

### 7.5 กฎการกรอก + Reconcile (เพิ่มจากแพร 2026-07-15)

**ก) กรอกหน่วยเต็ม + เศษ pack (P0-11)**
รายการที่ขายเป็นแพ็ค/กล่องแต่เปิดใช้ทีละส่วน ต้องกรอกได้ **2 ค่าคู่กัน** — จำนวนเต็ม (ที่ยังไม่เปิด) + เศษที่เหลือในแพ็คที่เปิดค้าง:

| รายการ | กรอกช่อง "เต็ม" | กรอกช่อง "เศษ" | หมายถึง |
|---|---|---|---|
| Greek Yogurt 1kg | `1` (box) | `1500` (g) | เหลือ 1 กล่องเต็ม + เปิดอีกกล่องเหลือ 1,500g |
| Plain Yogurt 500g | `2` (box) | `300` (g) | 2 กล่องเต็ม + เศษ 300g |

- ระบบเก็บ **2 หน่วยแยกกัน** (แพ็ค + กรัม/ชิ้น) — carry-forward + คำนวณคงเหลือทั้งสองหน่วยเป็นอิสระ (ตรงกับ StockDaily ที่มี ยกมา/คงเหลือ ทั้งชิ้นและกรัมอยู่แล้ว)
- ✅ กรณีนี้ **โครงข้อมูลเดิมรองรับแล้ว** — เพิ่มแค่ทำให้ **ช่องกรอกบนมือถือชัดว่าเป็น "เต็ม | เศษ"** ต่อรายการที่มีหน่วยผสม

**ข) Reconcile ถ้วยเสิร์ฟสิ้นวัน (P0-12)** — ถ้วยที่ลูกค้าใช้: **Cup P (5oz) · Cup S (9oz) · Small Bowl · Cup (14oz)**
เป้าหมาย: จำนวนถ้วยที่ **ใช้จริง** (นับกาย) ต้องตรงกับจำนวนถ้วยที่ **ขายไปจริง** (ตาม order)

```
ถ้วยใช้จริง (ต่อขนาด)  = ถ้วยตั้งต้นวัน + รับเข้า − คงเหลือสิ้นวัน
Reconcile (ต่อขนาด)     = ถ้วยใช้จริง − ถ้วยที่ขายตาม order
Reconcile (รวมทุกขนาด)  = Σถ้วยใช้จริง − Σถ้วยที่ขาย   ← เผื่อกรณีสลับขนาด
```
- ⚠️ **ปัญหาที่ต้องรองรับ:** พนง.อาจหยิบถ้วย**ขนาดไม่ตรง order** (เช่น order Cup P แต่ใช้ Cup S) → รายขนาดจะเพี้ยน (ขนาดนึงเกิน อีกขนาดขาด) แต่**ยอดรวมยังตรง**
- ระบบต้องเช็ก **2 ระดับ:** (1) รายขนาด — flag ถ้าต่าง, (2) กลุ่มรวม — ถ้ารวมตรงแต่รายขนาดเพี้ยน → mark ว่า "น่าจะสลับขนาด" (เตือนเบา ไม่ใช่ error หนัก)
- มีแถว `🔢 รวมถ้วยทั้งหมด` ในฟอร์มเดิมอยู่แล้ว → ใช้เป็นฐานของ reconcile รวม
- ✅ **เคาะแล้ว (แบบ ก, 2026-07-15):** "ถ้วยที่ขายจริงต่อขนาด" ให้ **พนง.กรอกเอง** ต่อขนาด ตอนปิดร้าน (v1) — ไม่พึ่ง POS/BOM · การ derive อัตโนมัติจากยอดขายเมนู (map เมนู→ขนาดถ้วย) ยกไป **Phase 3** (P2-5)

### 13.1 Config โหมดขายต่อ item (P0-13)

### 13.2 UOM + Sale Unit model (แกะกล่อง) (P0-14, P0-15)
- รายการแกะ = **UOM (แพ็ค/กล่อง เต็ม) + Sale Unit (เศษ นับเป็น g หรือ ชิ้น)** · แปลงด้วย `จำนวน/แพค`
- **เศษคงเหลือเกินเมื่อวานได้** (แกะกล่องใหม่ = ย้ายกรัมจากกล่องเข้าเศษ)
- **เช็กยอดรวม:** `คงเหลือรวม (แพ็ค×N + เศษ) ≤ ของที่มี (ยกมา+รับเข้า)` → เกิน = เตือน "คงเหลือรวมเกินของที่มี"
- **กรอกสองทาง:** ขาย/ใช้ ↔ คงเหลือ (ทั้งแพ็คและเศษ) — กรอกช่องไหน อีกช่องคำนวณให้

### 13.3 กลุ่มเศษรวม — สินค้าหลายขนาดที่เศษปนกัน (P0-16)
- **ปัญหา:** Strawberry (250g/500g), Blueberry (125g/300g/500g) เมื่อแกะแล้วเศษปนกัน บอกไม่ได้ว่ามาจากกล่องไหน
- **แก้:** จัด **กลุ่มเศษรวม** (ตั้งหน้า Settings) — แสดงทุกขนาด, แต่ **เศษกรอกได้ที่ item เดียว (leader)** กันกรอกซ้ำ (ขนาดอื่นขึ้นโน้ต "กรอกที่ …")
- แต่ละขนาดมี gram/กล่อง (250/500…) → เช็ก **ยอดรวมกลุ่ม (Σกล่อง×ขนาด + เศษ) ≤ ของเมื่อวาน**

### 13.4 ถ้วยเสิร์ฟ — นับรวม + ดึงจาก stock (P0-12, P0-17)
- Cup P/S/Small Bowl/Cup 14oz = แกะ **50 ชิ้น/แพ็ค** → หน้าสต็อกแสดง **"รวมทั้งหมด X ชิ้น (บันทึกวันนี้)"** ตลอด
- หน้า **Reconcile ถ้วย** `/cups`: **ตั้งต้น/รับเข้า/คงเหลือ ดึงจากยอดถ้วยหน้าสต็อกอัตโนมัติ** (แพ็ค×50 + เศษ) → พนง.กรอกแค่ **ขายจริง** · ระบบจับ "สลับขนาด" (รวมตรงแต่รายขนาดเพี้ยน)

### 13.5 Par + ส่งคืน/เสีย ในหน้ากรอกสต็อก (P0-18, P0-19)
- ทุก item โชว์ badge **`Par X`** (เป้าขั้นต่ำของสาขา)
- ช่อง **ส่งคืน/เสีย** → หักจากคงเหลือ (stock) ทันที · ถ้ากรอก > 0 เปิดช่อง **หมายเหตุ** ให้ใส่เหตุผล (หมดอายุ/แตก) · บันทึกลง `stock_daily.returned` + `note`

- **getRestock คงเหลือ = 0 บน Supabase** → เปลี่ยนไปคิดคงเหลือผ่าน `getStock` (carry-forward)
